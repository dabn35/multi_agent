## 4.1 왜 API를 MCP 서버로 감싸는가

- 외부 API는 서비스마다 사용법이 전혀 다르다
  - 예: 날씨 API는 `?q=Seoul&appid=KEY` 쿼리 파라미터를 쓰지만, GitHub API는 `Authorization: Bearer TOKEN` 헤더와 `/repos/owner/name` 경로를 쓴다
  - URL 구조, 인증 방식, 응답 JSON 형태가 모두 다르므로 매번 새로 배워야 한다
- 에이전트가 외부 API를 직접 다루면 다음 문제가 발생한다
  - 인증 키 노출 위험
  - 에러 처리 방식이 제각각임
  - 같은 작업을 여러 프로젝트에서 반복 구현하게 됨
- MCP 서버로 감싸면 얻는 이점
  - 에이전트는 단순한 도구 인터페이스만 사용 — 예: `get_weather(city="Seoul")`만 호출하면 되고, URL 구조나 인증 헤더를 몰라도 됨
  - 인증과 에러 처리를 서버 안으로 숨길 수 있음 — API 키는 서버 환경변수에만 존재하므로 대화 로그에 노출되지 않음
  - **한 번 만들면 여러 사람, 여러 에이전트가 그대로 쓸 수 있음** — `.mcp.json`에 한 줄 추가하면 어떤 프로젝트, 어떤 에이전트에서든 동일한 도구를 바로 호출할 수 있다. 팀원이 각자 API 연동 코드를 작성할 필요가 없어진다

### 왜 에이전트 구축에 MCP 서버가 필요한가

다음 주(Week 5)에는 도구를 직접 골라서 호출하는 에이전트를 만든다. 에이전트가 도구를 쓰려면 **잘 만들어진 도구**가 먼저 있어야 한다.

- 이번 주에 만드는 MCP 서버 = 에이전트가 사용할 **도구**
- 다음 주에 만드는 에이전트 = 그 도구를 **선택하고 호출**하는 주체

도구의 설명이 모호하거나, 에러 처리가 없거나, 입출력 형식이 제각각이면 에이전트는 도구를 제대로 고를 수 없다. 이번 주에 "설명 가능한 구조"를 강조하는 이유가 여기에 있다.

**한 줄 정리**

MCP 서버는 에이전트에게 도구를 제공하는 부품이다. 좋은 부품이 있어야 좋은 에이전트를 만들 수 있다.

## 4.3 실습: 날씨 MCP 서버 만들기

### 실습 목표

- 도시 이름을 입력받아 현재 날씨를 반환하는 MCP 도구를 만든다
- 입력: 도시 이름 — 출력: 온도, 습도, 날씨 상태
- 예외 처리: 잘못된 도시명, 인증 실패, 타임아웃

### 실습 위치

- 실습 작업 폴더: `practice/lectures/week-04/`
- 참고 예제 폴더: `practice/chapter4/`

### 바이브 코딩 진행 방식

한 번에 긴 프롬프트를 던져 완성품을 받는 방식보다, 아래처럼 짧은 단계로 나눠 Copilot과 대화하며 만드는 편이 좋다.

### 단계 1. 프로젝트 뼈대 만들기

Copilot에 요청할 것:

```text
practice/lectures/week-04 폴더에
Python 기반 MCP 날씨 서버 실습 프로젝트를 만들고 싶다.
다음 파일부터 만들어줘.
- requirements.txt
- .env.example
- .gitignore
- weather_server.py

조건:
- OpenWeatherMap API를 사용
- 실제 API 키는 .env로 읽기
- 지금은 함수 뼈대와 주석만 먼저 만들기
```

사용자가 확인할 것:

- 폴더와 파일이 생성되었는가
- `.env.example`에 실제 키가 들어가지는 않았는가
- `weather_server.py`가 너무 많은 기능을 한 번에 넣지 않았는가

### 단계 2. 외부 API 호출 함수 만들기

Copilot에 요청할 것:

```text
weather_server.py에
도시 이름(예: "Seoul")을 받아 OpenWeatherMap Current Weather API로
현재 날씨를 조회하는 함수 fetch_weather(city: str)를 만들어줘.

조건:
- requests.get()에 timeout=10을 넣어줘
- API 호출 함수와 응답을 정리하는 함수를 분리해줘
  - fetch_weather(): HTTP 요청만 담당
  - parse_weather_response(): 응답 JSON에서 온도·습도·날씨 상태만 뽑아 dict로 반환
- 실패 원인을 구분해줘
  - 404 → "도시를 찾을 수 없습니다"
  - 401 → "API 키가 유효하지 않습니다"
  - timeout → "서버 응답 시간 초과"
```

사용자가 확인할 것:

- API 호출 함수와 응답 정리 함수가 분리되었는가
- timeout이 들어갔는가
- 도시명 오류와 인증 오류가 구분되는가

### 단계 3. MCP 도구로 연결하기

Copilot에 요청할 것:

```text
방금 만든 날씨 조회 함수를
MCP 도구로 연결해줘.
입력은 city 문자열 하나만 받게 하고,
에러가 나면 JSON 형태로 설명을 반환하게 해줘.
```

사용자가 확인할 것:

- 도구 설명이 충분히 분명한가
- 입력이 단순한가
- 정상/실패 응답 형식이 일정한가

### 단계 4. 로그와 저장 추가하기

에이전트가 MCP 도구를 호출하면 그 과정은 눈에 보이지 않는다. 문제가 생겼을 때 "어디서 실패했는가"를 알 수 있는 유일한 단서가 로그다. 로그 없이는 디버깅이 불가능하고, 결과 저장 없이는 제출물도 만들 수 없다.

Copilot에 요청할 것:

```text
weather_server.py에
최소 로깅을 넣어줘.
조건:
- 요청 시작
- 성공
- 실패 원인
만 남기고, API 키는 로그에 남기지 않게 해줘.
또 결과를 output 폴더에 JSON으로 저장하게 해줘.
```

사용자가 확인할 것:

- 로그에 키가 보이지 않는가
- 결과 파일이 저장되는가
- 로그가 너무 장황하지 않은가

### 단계 5. 테스트 또는 검증 코드 추가하기

MCP 서버를 매번 띄워서 에이전트로 호출해 보는 것은 느리고 번거롭다. 핵심 함수(`fetch_weather`, `parse_weather_response`)만 따로 테스트할 수 있으면 MCP 서버 없이도 로직이 맞는지 빠르게 확인할 수 있다. 이 습관은 다음 주에 에이전트를 만들 때도 그대로 적용된다.

Copilot에 요청할 것:

```text
이 프로젝트에 간단한 테스트용 스크립트를 추가해줘.
목표:
- 정상 도시명 테스트
- 잘못된 도시명 테스트
- 입력 검증 테스트
MCP 서버 전체를 띄우지 않아도 핵심 로직을 확인할 수 있게 해줘.
```

사용자가 확인할 것:

- 테스트 코드가 API 호출 로직과 강하게 섞여 있지 않은가
- 핵심 함수만 따로 확인 가능한가
- 실패 테스트가 실제로 실패를 잡아내는가

### 실행 확인

모든 단계를 마친 뒤 아래 네 가지를 직접 돌려 본다.

- 정상 도시명으로 한 번 실행
- 없는 도시명으로 한 번 실행
- API 키 제거 상태로 한 번 실행
- 로그 파일 또는 출력 파일이 실제로 남는지 확인

## 4.4 실습: 외부 MCP 서버 활용하기

### 실습 목표

- 이미 만들어진 외부 MCP 서버를 `.mcp.json`에 등록하고 사용한다
- 자기가 만든 날씨 서버와 외부 서버를 **동시에** 연결해 본다
- "한 줄 추가로 도구를 공유한다"는 MCP의 핵심 가치를 체험한다

### 왜 외부 MCP 서버를 써 보는가

4.1에서 "한 번 만들면 여러 사람이 그대로 쓸 수 있다"고 했다. 외부 MCP 서버는 그 말의 실제 증거다. 누군가 만들어 둔 서버를 설정 한 줄로 내 에이전트에 붙일 수 있다.

직접 만든 서버와 외부 서버를 나란히 쓰면 다음을 느낄 수 있다.

- 도구 설명이 좋으면 사용법을 따로 배울 필요가 없다
- 설정만으로 에이전트의 능력이 확장된다
- 이번 주에 내가 만든 서버도 다른 사람에게는 "외부 서버"가 된다

### 사용할 외부 MCP 서버

아래 서버는 모두 **API 키가 불필요**하고, 한 줄로 설치된다.

#### 필수 (2개 모두 사용)

| 서버 | 설치 명령 | 하는 일 |
|---|---|---|
| **Filesystem** | `npx @modelcontextprotocol/server-filesystem /허용/경로` | 지정한 폴더의 파일을 읽고 쓴다. 허용 경로 밖은 접근 불가 |
| **Fetch** | `npx @modelcontextprotocol/server-fetch` | URL을 입력하면 웹 페이지를 마크다운으로 변환해 반환한다 |

#### 선택 (1개 이상 골라서 추가)

| 서버 | 설치 명령 | 하는 일 |
|---|---|---|
| **Memory** | `npx @modelcontextprotocol/server-memory` | 엔티티와 관계를 JSON 파일에 저장하는 지식 그래프 |
| **Playwright** | `npx @playwright/mcp@latest` | 브라우저를 자동으로 열어 클릭, 입력, 스크린샷 수행 |
| **DuckDuckGo Search** | `uvx duckduckgo-mcp-server` | 실시간 웹 검색 (API 키 없이 사용 가능) |

자유 선택으로 위 목록 외의 서버를 써도 된다. [MCP 서버 목록](https://github.com/modelcontextprotocol/servers)에서 찾을 수 있다.

### 진행 순서

#### 단계 1. `.mcp.json`에 외부 서버 등록

프로젝트 루트(`practice/lectures/week-04/`)에 `.mcp.json` 파일을 만들고, 자기가 만든 날씨 서버와 외부 서버를 함께 등록한다.

```json
{
  "mcpServers": {
    "weather": {
      "command": "python",
      "args": ["weather_server.py"]
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "./output"]
    },
    "fetch": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-fetch"]
    }
  }
}
```

#### 단계 2. 외부 서버 호출 확인

에이전트에서 외부 MCP 도구를 실제로 호출해 본다.

- Filesystem: `output` 폴더에 저장된 날씨 결과 JSON을 읽어 보기
- Fetch: 특정 URL의 내용을 가져와 요약해 보기

#### 단계 3. 복합 사용

자기가 만든 날씨 서버와 외부 서버를 **한 대화 안에서 함께** 사용해 본다.

예시 시나리오:

```text
서울 날씨를 조회하고, 그 결과를 output 폴더에 저장한 뒤,
저장된 파일을 Filesystem 도구로 다시 읽어서 확인해줘.
```

이 과정에서 에이전트가 여러 도구를 자연스럽게 조합하는 것을 관찰한다.

### 확인할 것

- `.mcp.json`에 서버가 3개 이상 등록되었는가
- 외부 서버 호출이 실제로 동작하는가
- 자기 서버와 외부 서버를 한 대화에서 함께 사용할 수 있는가

## 4.5 바이브 코딩에서 꼭 막아야 할 것

Copilot이 빠르게 코드를 만들어 주더라도 아래는 사용자가 직접 점검해야 한다.

- 한 파일에 모든 로직을 몰아넣지 않았는가
- API 키를 코드나 로그에 남기지 않았는가
- 타임아웃이 빠지지 않았는가
- 에러를 전부 `except Exception` 하나로 덮어버리지 않았는가
- 성공 사례만 보고 끝내지 않았는가

이번 주의 목표는 "한 번 돌아가는 코드"보다 **설명 가능한 구조**를 만드는 것이다.

## 4.6 API 키 관리

### 잘못된 방식

- 코드에 API 키를 직접 넣는 방식
- 공개 저장소 업로드 시 즉시 노출될 수 있음

```python
api_key = "hard-coded-secret"
```

### 권장 방식

- `.env` 파일에 키 저장
- `.env.example`로 필요한 변수 이름만 공유
- 코드에서는 `os.getenv()`로 읽기

```python
from dotenv import load_dotenv
import os

load_dotenv()
api_key = os.getenv("OPENWEATHERMAP_API_KEY")

if not api_key:
    raise ValueError("OPENWEATHERMAP_API_KEY가 필요합니다.")
```

### 체크포인트

- `.env`는 `.gitignore`에 들어가 있는가
- `.env.example`에는 실제 키가 없는가
- 키가 없을 때 친절한 오류를 보여 주는가

## 4.7 실패 처리를 왜 넣어야 하는가

외부 API는 항상 정상 동작하지 않는다.

- 네트워크가 느릴 수 있음
- 잘못된 도시명을 넣을 수 있음
- 키가 잘못되었을 수 있음
- 서버가 일시적으로 응답하지 않을 수 있음

이번 주 실습에서는 아래 세 가지만 반드시 챙긴다.

### 1. 타임아웃

```python
response = requests.get(url, timeout=10)
```

- 무한 대기를 막는다

### 2. 재시도

- 모든 오류를 무한 재시도하지 않는다
- 일시적 네트워크 오류만 1~2회 다시 시도한다

### 3. 명확한 에러 메시지

- "실패"라고만 하지 말고 원인을 구분한다
  - 인증 실패
  - 입력 오류
  - 네트워크 오류
  - 타임아웃

---

## 4.8 최소 로깅

이번 주 로깅은 운영 수준이 아니라 디버깅 가능한 최소 수준이면 충분하다.

기록할 것:

- 어떤 입력이 들어왔는지
- 외부 API 호출이 성공했는지
- 실패했다면 어떤 종류의 실패인지

기록하지 말 것:

- API 키
- 민감한 전체 URL
- 필요 이상으로 긴 원본 응답 전체

```python
logger.info("weather request started: city=%s", city)
logger.warning("weather request failed: city=%s", city)
```

## 4.9 테스트 가능한 구조

실습에서 가장 많이 생기는 문제는 "API 호출 코드와 비즈니스 로직이 한 함수에 섞이는 것"이다.

권장 구조:

1. 외부 API를 호출하는 함수
2. 응답을 정리하는 함수
3. MCP 도구로 연결하는 함수

이렇게 나누면 좋은 점:

- API 없이도 응답 정리 함수를 테스트할 수 있음
- 에러 상황을 독립적으로 점검할 수 있음
- MCP 도구 바깥에서 먼저 디버깅할 수 있음

## 제출물

### A. 직접 만든 MCP 서버

- MCP 서버 코드
- 실행 결과 캡처 2장 이상 (정상 + 실패)
- 사용한 핵심 프롬프트 3개 이상
- 짧은 메모
  - 어떤 에러 처리를 넣었는가
  - 로그에 무엇을 남겼는가
  - Copilot이 제안한 것 중 무엇을 수정했는가

### B. 외부 MCP 서버 활용

- `.mcp.json` 설정 파일 (자기 서버 + 외부 서버가 함께 등록된 상태)
- 외부 MCP 서버 호출 결과 캡처 2장 이상
- 비교 메모 (3~5줄)
  - 직접 만든 서버와 외부 서버의 사용 경험 차이
  - 예: 설정만으로 도구가 추가되는 편리함, 도구 설명의 중요성, 여러 도구를 조합했을 때 느낀 점

## 체크리스트

### 직접 만든 서버

- 빈 폴더 또는 새 실습 폴더에서 시작했다
- `.env`와 `.env.example`를 구분했다
- API 키를 코드에 직접 쓰지 않았다
- 타임아웃을 설정했다
- 입력 오류와 네트워크 오류를 구분했다
- 최소 한 개 이상의 로그를 남겼다
- 정상 실행과 실패 실행을 모두 확인했다
- Copilot이 만든 코드를 그대로 믿지 않고 직접 읽어 봤다

### 외부 서버 활용

- `.mcp.json`에 자기 서버와 외부 서버가 함께 등록되어 있다
- 외부 서버를 최소 2개 이상 사용했다 (필수 2개 + 선택 1개 권장)
- 외부 서버 호출 결과를 캡처했다
- 자기 서버와 외부 서버를 한 대화에서 함께 사용해 봤다
- 비교 메모를 작성했다
