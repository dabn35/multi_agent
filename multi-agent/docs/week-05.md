## 5.1 에이전트는 어떻게 도구를 고르는가

- 에이전트는 마법처럼 도구를 고르는 것이 아니다
- 실제로는 모델이 도구의 이름, 설명, 입력 형식을 보고 선택한다

도구 선택이 흔들리는 대표적인 경우:

- 설명이 모호함
- 비슷한 도구가 여러 개 있음
- 입력 형식이 불분명함

좋은 도구 설명의 조건:

- 무엇을 하는지 분명해야 함
- 언제 써야 하는지 분명해야 함
- 어떤 입력을 받는지 보여 줘야 함

---

## 5.2 도구 설명을 잘 써야 하는 이유

예를 들어 `get_weather`와 `get_coordinates`가 함께 있다면 모델은 다음을 알아야 한다.

- 도시 이름만 있을 때는 먼저 좌표를 구해야 함
- 좌표가 있으면 바로 날씨를 조회하면 됨

즉, 도구 설명은 사람이 읽는 문서이면서 동시에 모델이 읽는 실행 규칙이다.

```python
@tool
def get_weather_tool(latitude: float, longitude: float) -> str:
    """위도와 경도를 받아 현재 날씨를 조회한다."""
```

---

## 5.3 입력 검증

에이전트가 만든 입력은 항상 올바르지 않다.

자주 발생하는 문제:

- 필수 값 누락
- 타입 오류
- 범위 오류

이번 주에는 Pydantic 같은 검증 도구를 사용해 최소한 아래를 확인한다.

- 위도와 경도 타입
- 허용 범위
- 필수 필드 존재 여부

```python
class WeatherInput(BaseModel):
    latitude: float = Field(..., ge=-90, le=90)
    longitude: float = Field(..., ge=-180, le=180)
```

---

## 5.4 구조화된 출력

- 모델 응답을 자유 텍스트로만 받으면 파싱 오류가 자주 발생한다
- JSON 형식이나 스키마를 강제하면 후처리가 쉬워진다
- 이번 주에는 "왜 필요한가"를 이해하고, 간단한 예시를 적용하는 수준이면 충분하다

실습 목표:

- 모델이 `{ "city": "...", "action": "..." }` 같은 구조를 따르게 만들기

## 5.6 실습: 날씨 조회 에이전트

### 실습 목표

- 2~3개의 도구를 연결해 간단한 에이전트를 만든다

### 권장 도구 구성

- 도시 이름을 좌표로 바꾸는 도구
- 좌표 기반 날씨 조회 도구
- 결과를 파일로 저장하는 도구

### 실습 흐름

1. 사용자 요청 입력
2. 좌표 조회 도구 호출
3. 날씨 조회 도구 호출
4. 결과 저장
5. 최종 응답 생성

### 학생이 직접 확인할 것

- 도구 선택 순서가 타당한가
- 잘못된 입력에서 어떻게 실패하는가
- 저장된 결과가 사람이 읽기 쉬운가
