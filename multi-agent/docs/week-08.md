## 8.1 7주차 미니 프로젝트 리뷰

각자 작성한 `weather_tool.py`, `agent.py`를 짝과 교환해 다음을 점검한다.

| 점검 항목 | 통과 기준 |
|---|---|
| 도구 docstring | `get_weather`가 *언제 쓰는지* 설명되어 있는가 |
| 상태 흐름 | `WeatherState`의 각 필드가 흐름을 반영하는가 |
| 분기 종료 | 무한 루프 없이 종료되는가 (`attempt >= 3`) |
| 실패 처리 | 잘못된 도시명에서 오류가 *드러나는가*, 숨겨지는가 |
| 설명 가능성 | 코드만 보고 무엇을 하는지 알 수 있는가 |

자주 본 실수:
- `attempt`를 0이 아니라 빈 문자열로 초기화 → 비교 오류
- LLM 응답을 `result.content`로 꺼내지 않고 객체 그대로 저장
- 검증 노드가 항상 “yes”만 답함 → 무한 루프 직전까지 감
- 존재하지 않는 도시에 대해 `KeyError`로 그대로 죽음

## 8.2 1~7주차 핵심 정리

| 주 | 핵심 한 줄 |
|---|---|
| 1 | AI가 코드를 써도 *사람의 검증 책임*은 남는다 |
| 2 | 재현 가능한 실행 환경이 모든 실습의 기반이다 |
| 3 | MCP의 3대 프리미티브는 Tool·Resource·Prompt다 |
| 4 | 외부 API 래핑은 인증·실패·로깅·테스트의 4박자다 |
| 5 | LangChain 에이전트는 도구 설명을 보고 도구를 고른다 |
| 6 | LangGraph는 상태·분기·반복을 *명시적으로* 설계한다 |
| 7 | MCP·도구·LangGraph가 한 시스템에서 결합된다 |

## 8.3 단일 에이전트의 한계 발견

7주차 코드를 다시 본다. 다음 질문에 답해 본다.

1. `agent.py`의 한 함수가 *여러 책임*을 동시에 지고 있는가?
2. 초안 생성과 검증을 *같은 모델 한 번*에 했더라면 더 나았을까?
3. 이 시스템이 *3배 복잡한 작업*을 받으면 어디서 먼저 깨질 것인가?
4. 만약 “비전공자 대상” 외에 “전문가 대상” 요약도 만들어야 한다면?

이 질문들에 답하다 보면 자연스럽게 “**이 작업을 두 명에게 나누면 어떨까**”라는 발상이 나온다. 그게 9주차의 출발점이다.

## 8.4 사전 메모: 자기 도메인의 멀티에이전트 후보

다음 양식을 채워서 9주차에 가져온다.

```markdown
## 멀티에이전트 사전 메모

- 내 관심 도메인:
- 그 도메인에서 자주 하는 작업 1개:
- 그 작업을 단일 에이전트(또는 사람 혼자)로 할 때 막히는 지점 (3줄):
- 만약 두 명/세 명에게 나눈다면 역할 분담 (각 1~2줄):
- 이 작업이 멀티에이전트로 더 나아질 것이라 보는 근거 1줄:
- 멀티에이전트가 *오히려 손해*일 수도 있는 이유 1줄:
```

마지막 줄(*손해일 수도 있는 이유*)이 가장 중요하다 — 9주차의 “언제 쓰지 말 것인가” 토론의 출발점이다.

## 8.5 활동: 짝 점검 + 사전 메모 작성

### 1회차 (이론·리뷰, 90분)

- 0~30분: 7주차 미니 프로젝트 짝 점검 (위 8.1 표 채우기)
- 30~60분: 1~7주차 핵심 정리 (8.2 표를 보지 않고 스스로 채워 보기)
- 60~90분: 9주차 이후의 흐름 미리 보기 — 강의계획서 9~13주차 표 함께 읽기

### 2회차 (실습 + 사전 메모, 90분)

- 0~30분: 7주차 미니 프로젝트 개선 (피드백 반영)
- 30~75분: 단일 에이전트 한계 분석 (8.3 질문 4개에 답하기)
- 75~90분: 사전 메모 작성 (8.4 양식 채우기)

## 다음 주 예고

9주차에 본격적으로 멀티에이전트로 들어간다. 단일 vs 멀티 비교를 *코드 위에서 직접* 해 보고, 4대 패턴(순차·계층·협력·시장)을 정리한 뒤, LangGraph로 2-에이전트 핸드오프를 만든다. 오늘 작성한 사전 메모가 9주차 활동의 출발점이 된다.
